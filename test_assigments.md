# QA Pre-Release Assessment — 24h to Release

---

## 1. Первые 30 минут

Сначала масштаб, потом детали.

1. Метрики нового билда vs предыдущий — crash rate, payment funnel drop-off, когорта новых установок отдельно
2. Воспроизвожу руками в порядке приоритета: платёж (A) → медиа (C) → чистая установка (B)
3. Смотрю diff и changelog по затронутым модулям — часто баг = "что-то недавно переписали"
4. Network через Charles/Proxyman — таймауты, пустые ответы, странные статусы
5. Фиксирую картину: что подтвердилось, что нет, где нет данных совсем

---

## 2. Факты, гипотезы, чего не хватает

| Сигнал | Гипотеза | Чего не хватает |
|--------|----------|-----------------|
| A (платежи) | Продукты не грузятся из API или UI завис в неожиданном state | % affected users, шаг воронки, логи SDK |
| B (новые установки) | Race condition: UI рендерится до получения auth token | Логи первой сессии, network waterfall, new vs returning code path |
| C (медиа) | Слишком широко для гипотезы — CDN, DRM, кодек, permissions, OS regression | Тип медиа, девайс, OS, конкретные шаги |
| D (UI) | Верстка не адаптирована под новые устройства — Dynamic Island, новые aspect ratio | Модели из жалоб, затронутые экраны, UI изменения в этом билде |
| E (навигация) | Скорее UX-изменение без коммуникации, чем баг. Или данные не мигрировали | Была ли переработка навигации, есть ли migration script |

## 3. Приоритизация

**P0 — блокируют релиз:**
- A (платежи) — прямые потери revenue прямо сейчас
- C (медиа) — если это core use case, приложение теряет смысл

**P1 — управляемый риск:**
- B (новые установки) — пользователь с "вечной загрузкой" в первой сессии просто удалит приложение
- D (UI на флагманах) — флагманы это платящая аудитория, баги там конвертируются в плохие отзывы

**P2 — не блокирует:**
- E (навигация) — раздражает, но core use case не сломан

---

## 4. Разбор сигналов

**A — Платежи**

Два проявления из жалоб — возможно две разных точки отказа:
- "пусто" — продукты не загрузились
- "не нажимается" — UI завис в неожиданном state

Маршрут:
1. Paywall открылся → есть продукты? Если нет — сразу в network tab и логи SDK
2. Sandbox-покупка end-to-end — без этого фикс не считается закрытым
3. Если код чистый — проверяю конфигурацию в App Store Connect / Google Play Console

Из практики: половина платёжных багов — это конфигурация стора, не код.

---

**B — Новые установки**

Маршрут:
1. Чистая установка → смотрю network waterfall: что именно висит — auth, config или контент
2. Проверяю: рендерится ли UI до получения auth token
3. Throttling сети — race condition начинает воспроизводиться стабильнее

---

**C — Медиа**

Первое — понять что именно не работает: воспроизведение, запись или стриминг. Без этого диагностика вслепую.

Маршрут:
1. URL контента отдаётся? (200 OK или 403/DRM error)
2. Ошибки плеера в логах
3. Permissions — запрашиваются ли и в правильный момент
4. Тест на разных OS версиях

Пока нет стабильного repro — код не трогаю.

---

**D — UI**

1. Фиксирую: какие экраны ломаются, а какие нет
2. Safe area insets, hardcoded размеры, Dynamic Type
3. Смотрю что менялось в UI-слое в этом билде

Чаще всего — быстрый layout fix без риска для бизнес-логики.

---

**E — Навигация**

1. Раздел существует в новом билде?
2. Обновляюсь с предыдущей версии — данные сохраняются?
3. Если раздел просто перемещён — UX-изменение без коммуникации, не баг

---

## 5. Использование LLM

Использую как инструмент ускорения, не как источник решения.

Помогает: анализ логов и stack traces, разбор diff, генерация edge cases, оформление баг-репортов.

Не использую: go/no-go решение, верификация фикса на девайсе, оценка бизнес-риска.

Любой вывод LLM проверяю руками — это не недоверие, это дисциплина.

---

## 6. Три сигнала, которые не закрываются с первой итерации

**A — Платежи (2–3 итерации)**

Первый фикс закроет одно проявление, но не оба. LLM не видит конфигурацию стора и runtime-состояние SDK — только код. После каждого фикса нужна sandbox-верификация руками.

Что делает человек: запускает тестовую покупку, проверяет конфиг в консоли стора, верифицирует на 2+ OS.

---

**B — Онбординг (2–3 итерации)**

Race condition воспроизводится непредсказуемо — зависит от timing и порядка инициализации. LLM предложит очевидный путь, но не угадает все состояния без полного контекста архитектуры.

Что делает человек: воспроизводит на чистом девайсе с throttling, анализирует waterfall, проверяет feature flags для новых пользователей.

---

**C — Медиа (не закрыть за 5 итераций)**

Проблема не в количестве итераций, а в отсутствии репро. LLM будет поочерёдно закрывать разные гипотезы — DRM, CDN, кодек, OS regression — но после каждой итерации нет критерия "сработало". Можно сделать 5 правок и не знать, помогло ли хоть одно.

Если причина в DRM — нужен вендор или DevOps с доступом к лицензионному серверу. Если OS regression — Apple/Google bug report или откат API. Если CDN — это вообще не code fix.

**Что должно произойти вне контура LLM:**
- Саппорт собирает чёткий repro: девайс, OS, тип медиа, шаги
- Разработчик, знающий медиа-стек, проводит диагностику с конкретным кейсом

Реалистичный путь: 1–2 недели с правильным человеком и полным контекстом. До релиза — feature flag off или блокировка релиза.

---

## 7. Stop-loss

Правило: **2 итерации без воспроизводимого подтверждения на реальном билде — останавливаюсь и эскалирую.**

| Условие | Действие |
|---------|----------|
| Фикс A не подтверждён в sandbox после 2 попыток | Блокирую релиз |
| Фикс C без стабильного repro | Feature flag off или блокировка |
| Фикс D частично работает | Релиз + hotfix P1 в 24h |
| Фикс B нестабилен | Blocker, не пускаю |

Каждое решение фиксирую письменно с timestamp.

## 8. Regression checklist

**Платежи:**
- Paywall открывается с продуктами
- Sandbox-покупка проходит end-to-end
- При ошибке — корректный fallback, не белый экран

**Медиа:**
- Core функция запускается без ошибки
- Воспроизведение стабильно 30+ секунд
- Permissions запрашиваются корректно

**Новый пользователь:**
- Чистая установка → контент появляется без вечной загрузки
- Auth завершается без зависания

**UI:**
- Главный и платёжный экран корректны на iPhone 15 Pro и Samsung S24
- Ключевые кнопки не обрезаются

**Общее:**
- Нет новых crash в логах после фикса

---

## 9. Технические решения

**A:** Async init StoreKit 2 / Billing Client с await перед рендером paywall. Explicit loading + error state с retry — не пустой экран.

**B:** Рендер контента только после completed auth + config fetch. Timeout с retry и понятным fallback вместо бесконечного спиннера.

**C:** Без стабильного repro — не трогаю. Любое изменение вслепую создаёт риск регрессии без гарантии фикса.

**D:** Hardcoded размеры → safe-area aware constraints. Проверить safeAreaInsets для Dynamic Island.

**E:** Если раздел перемещён — tooltip или redirect. Если данные не мигрировали — hotfix миграции обязателен до релиза.

---

## 10. Решение по релизу

| Статус | Сигнал | Условие |
|--------|--------|---------|
| Не пускать | A, C | Не подтверждены как исправленные на билде |
| Релиз + hotfix 24h | D | Не затрагивает платёжный флоу |
| Релиз + мониторинг | B | Метрики не показывают массовость |
| Следующий релиз | E | Данные на месте, только навигация изменилась |

**GO если:** A и C верифицированы, D не в платёжном флоу, нет новых P0 крашей.

**NO-GO если:** A или C не закрыты. Любой новый блокирующий краш в regression.

---

## 11. Сообщение для PM

Два критичных риска — платёжный флоу и медиа. Если они не работают стабильно: прямые потери по деньгам и retention.

Платёжку диагностирую — нужен доступ к sandbox и логам SDK. По медиа сигнал слишком размытый, без репро (девайс + OS + шаги) фиксить вслепую опасно. Онбординг и UI — управляемые риски, не блокируют. Навигация — следующий релиз.

GO только если A и C подтверждены на билде. Если к [дедлайн — 4 часа] подтверждения нет — отключаем фичи или переносим релиз.

Нужна помощь: кто из команды владеет медиа-стеком и может дать репро по сигналу C?
